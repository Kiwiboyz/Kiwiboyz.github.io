About

Whats the project?

The task of this project was to build a web app that allowed users to see the upcomming semester dates , this allows the user to find out important information such as public holidays, upcoming assignments and the current week of the semester

With a tool like this it should help both students and lectures plan out there semeseter more easily without the stress of forgotten dates

Summary

I will do a write up closer to the end of the project so far i belive we are on the right track as a team and feel good with the amount of work I have done

Professional proficiency

Communication

From day one our communication was on point, as soon as we were put into our groups we set up a Discord server and Gitlab. We chose Discord because it was easier to use, free, and most of us already had an account already set up. Discord also has the ability to handle voice chat so we could call into the group and have conference calls on the go. Although we didn’t end up making use of this feature The beginning of the Discord channel
Having communications, right from the get-go, set us up for a smooth start for the project We also had scrums every lesson we did try to have them everyday but due to the timetable constraints we weren’t always around Polytechnic, so we couldn’t follow agile principle 6 “The most efficient and effective method of conveying information to and within a development team is a face-to-face conversation.” in the case, we couldn’t make it we would usually give the team a quick update on the Discord channel. Here we are talking about the user stories we had recently collected


We used other tools to communicate like the bug tracking system this was great because we could see what other members of the team were currently working on and what version of the software we were up to. This helped us organize our sprints weekly by saying that the tasks assigned would be completed by the end of the week. In some cases, our sprints went for two weeks but that was about the max amount of time we spent on something before asking for help. An image of the bug tracking website Roadmap
We would usually send files to each other via the Discord files and images channel this allowed us to keep the main channel clear of clutter and keep everything organized. The files posted here were not supposed to be included in the main project so they were not uploaded to the Gitlab for this reason

This year communication was a primary focus, so I ended up sending what felt like chapters of text to the channel to keep everyone informed on either what I was doing or what went wrong. This turned out to be a lot of help because when we got back into class we didn’t have to spend 20 or so minutes discussing what had happened.

Reflection

We could have communicated a bit more with our Clients as this would have allowed us to add more to the project although we got a lot of work done considering we started from scratch.



Leadership

This Year I took more of a management and leadership role I had learnt from previous experiences that taking charge instead of just sitting there is definitely a better option I started out by getting our group to do a quick introductory talk just saying the first name and what they like to do, I did this so we would know each other's names. I then created a Discord group and added everyone so we had that channel open,
I wanted to have some sort of software to properly track what everyone was doing so we could provide good data for the mid and end project progress reports I decided to use mantis BT. An open source free bug and issue tracker that I ran on my personal servers.

The best thing about this was that it was open source so I could easily make modifications to it to suit our needs I added a user stories Row so we could specify what was a User story

Most of the team members used it after they got used to it. This made it great because we all knew who was working on what. In Week 2 I tried to organize the group by splitting it into to parts our frontend team and the backend team but this quickly changed after the initial version of the database was released. I also helped out on the front end side but I was trying to keep everything organized for the team.

Reflection

Looking back, I think I did a good job in leading our team. Making sure everyone was working on something or was going to do something as well as keeping good records on what had happened. If there was any self-advice I’d suggest to make sure to write down what happened in conversations and team scrums rather than trying to commit them to memory.



Teamwork

With the Bug tracking software and the weekly sprints I believe we communicated really well there were times when we didn’t know what team members were Woking on but this came down to them not entering it in to Mantis, The way we fixed this was just talk to them and ask them to add it so we knew what was happening
If we ever ran into an issue we would ask other members and they always seemed happy to help this helped a lot especially around learning the whole YII framework.
An example of what really showed our teamwork was when Aidyn worked out how to use it and explained to me that we had a page generator built into the Framework named Gii
This really ended up coming in handy as now we had an easy way to create views and CRUD fields. I really enjoyed working in a team that communicated well and contributed to the project.

Reflection

If I was to try and improve anything it would be to make sure that we knew what the rest of the team members were doing in some cases, we were not sure of what someone was working on and if they could have been working on something more aligned with the user stories


Time management

I tried to attend the class as much as possible although due to other commitments id have to skip it which I found annoying because I always liked to do a scrum with the rest of the team members, I’d always ask the team later on the discord what happened.
When an issue arose I’d usually try to get to it as soon as possible. This was highly depended on what the issue was for example if it was a simple database import error then id gets to it within the day but if it was something more serious like the website being down then id get to it as soon as possible.
When the Mantis site went down I spent about an hour trying to work out what went down and how to fix it. Because I had automated logging active it was only down for about an hour So I was able to get to it up and running pretty quickly.

Mantis BT also had a roadmap that allowed us to set up versions this let us plan our sprints around it, as one of our main goals was following an agile approach, in this case, we wanted to satisfy the customer through the early and continuous delivery of valuable software which is, in fact, the first Agile principle


Reflection

I felt that I managed my time relatively well considering I had my other classes and commitments I was able to keep contributing to the project and keep everything online (for the most part) throughout the project. If I was to improve anything it would be devoting more time towards the project.


Motivating self and others

Throughout the project I was continually asking team members to add tickets to the bug tracker, so we could track the project. This worked out well because we were able to see common issue trends usually these were simple design mistakes.
My motivation was from failing last semester so this year I went all out and put a lot more effort into this paper recording and collecting data on almost everything. I worked a lot more with the team and communicated with them on what I was doing. I tried to provide as much help as possible to all team members. Because of this, I am actually quite happy with what I achieved this semester.

Reflection

Looking back, I Think I did a pretty good job I definitely had the motivation with not wanting to repeat again and feel like I was able to keep the team motivated with showing them what they had done for the mid project report. Overall, I don’t have any improvement advice for this.
